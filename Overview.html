Goals

  • Define the finite list of encodings and labels for the web platform so the "CHARACTER SETS" registry is no longer needed for it.
  • Define error handling for the encodings.
  • Mark encodings that are not utf-8 as legacy.

Notes

  • Labels defined for each encoding is a conservative list at this point. We can extend it as we please provided there is some agreement to do so.
  • Mapping for each encoding is done by majority of implementations. If the majority wants to change or has to change for compatibility, we can change the mapping.

1 Preface

While encodings for the web platform have been defined to some extent, implementations have not always implemented them in the same way, have not always used the same labels, and often differ in dealing with undefined areas of encodings. This standard attempts to fill those gaps so that new implementations do not have to reverse engineer encoding implementations of the market leaders.

2 Terminology

A code point is a Unicode code point and is referenced as a four-or-more digit hexadecimal value, typically preceded by "U+". E.g. U+0020.

The fallback code point is U+FFFD.

An octet is referenced as a double-digit hexadecimal value, typically preceded by "0x". E.g. 0xAA.

The space characters, for the purposes of this specification, are U+0020 SPACE, U+0009 CHARACTER TABULATION (tab), U+000A LINE FEED (LF), U+000C FORM FEED (FF), and U+000D CARRIAGE RETURN (CR).

Comparing two strings in an ASCII case-insensitive manner means comparing them exactly, code point for code point, except that the characters in the range U+0041 to U+005A (i.e. LATIN CAPITAL LETTER A to LATIN CAPITAL LETTER Z) and the corresponding characters in the range U+0061 to U+007A (i.e. LATIN SMALL LETTER A to LATIN SMALL LETTER Z) are considered to also match.

3 Encodings

An encoding defines a mapping from a code point to one or more octets (and vice versa). Each encoding has a name, and one or more labels.

Each encoding also has a to Unicode and from Unicode algorithm.

The to Unicode algorithm takes a stream of octets and emits a stream of code points. It has an associated fatal flag that when set will make the algorithm return failure if an invalid octet or octet sequence is in the stream. The octet pointer is initially zero, pointing to the first octet in the stream. It cannot be negative. It can be increased and decreased to point to other octets in the stream. The EOF octet is a conceptual octet representing the last octet in a stream. The octet pointer cannot point beyond the EOF octet. The EOF code point is a conceptual code point that is emitted once the to Unicode algorithm is done. As long as the EOF code point is not emitted, the to Unicode algorithm must be invoked for each code point emitted.

The from Unicode algorithm takes a stream of code points and emits a stream of octets. It will fail when a code point is passed for which it does not have a corresponding octet or octet sequence. Analogously to to Unicode, it has a code point pointer. Again analogously, as long as the EOF octet is not emitted, the from Unicode algorithm must be invoked for each octet emitted.

The fatal flag is for XML, which considers such an error a well-formed violation. Other formats such as HTML will use fallback code points for such errors and carry on.

Pre-processing is required to make from Unicode not fail (e.g. for HTML forms).

The table below lists all encodings and their labels user agents must support. User agents must not support any other encodings or labels.

To get an encoding from a string label, run these steps:

  1. Remove any leading and trailing space characters from label.

  2. If label is an ASCII case-insensitive match for any of the labels listed in the table below, return the corresponding encoding and terminate these steps.

  3. Return failure.

This algorithm is different from the one defined in section 1.4 of Unicode Technical Standard #22 as that algorithm is incompatible with legacy content.

New content and formats must exclusively use the utf-8 encoding.

Need to define decode a byte string as UTF-8, with error handling in a way that avoids external dependencies.

5 Legacy single-octet encodings

An encoding where each octet is either blank or a code point is a single-octet encoding. Encodings defined in subsections of this section are single-octet encodings.

In a single-octet encoding table the octet can be found by concatenating the row and column header respectively. The cell corresponding to those headers is either blank or a code point (with "U+" omitted for brevity).

In the single-octet encoding tables in this subsections rows where each octet's corresponds to a code point of the same value, are omitted.

The to Unicode and from Unicode algorithms are the same for all single-octet encodings, except for the table in use.

The to Unicode algorithm is as follows:

  1. Let octet be octet pointer.

  2. If octet is the EOF octet, emit the EOF code point.

  3. Increase the octet pointer by one.

  4. If octet is listed in the table, its corresponding cell is blank, and the fatal flag is set, terminate the overall algorithm and return failure.

  5. If octet is listed in the table and its corresponding cell is blank, emit a fallback code point.

  6. Emit a code point whose value is in octet's corresponding cell.

The from Unicode algorithm is as follows:

  1. Let code point be code point pointer.

  2. If code point is the EOF code point, emit the EOF octet.

  3. Increase the code point pointer by one.

  4. If code point is not listed in the table, terminate the overall algorithm and return failure.

  5. Emit code point's corresponding octet.

See also: single-octet research.

Single-octet encoding data is also available as a non-normative JSON resource: single-octet-encodings.json.

The euc-jp offset is for a given octet is null if octet − 0xA1 is greater than 93 or less than 0, or otherwise the result of that equation.

The to Unicode algorithm is:

  1. Let octet be octet pointer.

  2. If octet is the EOF octet, emit the EOF code point.

  3. Increase the octet pointer by one.

  4. If octet is in the range 0x00 to 0x7E, emit a code point whose value is octet.

  5. If octet is not 0x8E, 0x8F, or in the range 0xA1 to 0xFE, and the fatal flag is set, return failure.

  6. If octet is not 0x8E, 0x8F, or in the range 0xA1 to 0xFE, emit a fallback code point.

  7. Let second octet be octet pointer.

  8. If second octet is the EOF octet and the fatal flag is set, return failure.

  9. If second octet is the EOF octet, emit the fallback code point.

  10. If second octet is in the range 0x00 to 0x7E and the fatal flag is not set, emit a fallback code point.

  11. Increase the octet pointer by one.

  12. If octet is:

    0x8E

    1. If second octet is in the range 0xA1 to 0xDF, emit a code point whose value is 0xFEC0 + second octet.

    2. If the fatal flag is set, return failure.

    3. Emit the fallback code point.

    0x8F

    1. If second octet is in the range 0xA1 to 0xFE, let lead be the jis0212 index for second octet.

    2. If the fatal flag is set and either second octet is not in the range 0xA1 to 0xFE or lead is null, return failure.

    3. If either second octet is not in the range 0xA1 to 0xFE or lead is null, emit the fallback code point.

    4. Let trail octet be the octet pointer.

    5. If trail octet is the EOF octet and the fatal flag is set, return failure.

    6. If trail octet is the EOF octet or is in the range 0x00 to 0x7E, decrease the octet pointer by one and emit the fallback code point.

    7. Otherwise, increase the octet pointer by one.

    8. Let offset be the euc-jp offset for the trail octet.

    9. If offset is null and the fatal flag is set, return failure.

    10. If offset is null, emit the fallback code point.

    11. Let code point be japanese code point for lead plus offset.

    12. If code point is null and the fatal flag is set, return failure.

    13. If code point is null, emit the fallback code point.

    14. Emit code point.

    0xA1 to 0xFE

    1. Let lead be the jis0208 index for octet and offset the euc-jp offset for second octet.

    2. If the fatal flag is set and either lead or offset is null, return failure.

    3. If either lead or offset is null, emit the fallback code point.

    4. Let code point be japanese code point for lead plus offset.

    5. If code point is null and the fatal flag is set, return failure.

    6. If code point is null, emit the fallback code point.

    7. Emit code point.

The to Unicode algorithm is:

  1. Let octet be octet pointer.

  2. If octet is the EOF octet, emit the EOF code point.

  3. Increase the octet pointer by one.

  4. If octet is in the range 0x00 to 0x80, emit a code point whose value is octet.

  5. Let lead be the shift_jis lead octet index or code point for octet.

  6. If lead is less than 0xF8F0 or null, run these substeps:

    1. Let trail octet be octet pointer.

    2. If trail octet is the EOF octet and the fatal flag is set, return failure.

    3. If trail octet is the EOF octet, emit the EOF code point.

    4. Let offset be the shift_jis offset for trail octet.

    5. If either offset or lead is null and the fatal flag is set, return failure.

    6. If offset is null, decrease the octet pointer by one and emit a fallback code point.

    7. Increase the octet pointer by one.

    8. If lead is equal or more than 0xE000, emit a code point whose value is lead plus offset.

    9. Otherwise, if lead is null, emit a shift_jis fallback code point.

    10. Otherwise, let code point be japanese code point for lead plus offset.

    11. If code point is null and the fatal flag is set, return failure.

    12. If code point is null, emit a shift_jis fallback code point.

    13. Otherwise, emit code point.

  7. Otherwise, emit a code point whose value is lead.
